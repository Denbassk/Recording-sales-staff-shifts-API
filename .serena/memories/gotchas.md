# mem:gotchas — грабли, на которые легко наступить

## Лимиты карт
- `getEmployeeCardLimit` при ЛЮБОЙ ошибке или отсутствии связки сотрудник→тип лимита
  МОЛЧА возвращает `DEFAULT_LIMITS.STANDARD` (8700/7900/90%). Сотрудник без
  `card_limit_type_id` тихо считается «обычной картой» — это намеренный fallback, но
  скрывает ошибки данных. Если у кого-то «не тот» лимит — проверь связь в employees.
- `buildFotReport` НЕ использует индивидуальные лимиты — там `cardLimit = 16000` хардкодом.
  Расчёт ФОТ и расчёт зарплаты используют РАЗНЫЕ лимиты карты. Не путать.

## Числа и форматирование
- `formatNumber(num)`: null/undefined → `'0,00'`; не-число → `'0,00'`. Возвращает СТРОКУ
  в русском формате (пробел = разделитель тысяч, запятая = десятичная). Нельзя сравнивать
  как число и нельзя слать обратно в расчёты.
- Округление аванса — `Math.floor(x/100)*100` (ВНИЗ до 100 грн), затем `min(maxAdvance)`.
- Сверки сумм идут с допуском: 0.01 в `validatePayrollCalculation`, 1 грн в final-payroll.
  Не ужесточать без понимания — копеечные расхождения ожидаемы.

## Даты
- `validateDate` запрещает будущее (если `allowFuture=false`) и год < `MIN_YEAR (2024)`.
  `today` берётся с временем 23:59:59 — «сегодня» проходит.
- Дата выручки парсится из ИМЕНИ ФАЙЛА (`DD.MM.YYYY` или `DD.MM.YY`, год `YY`→`20YY`).
  Если в имени нет даты — используется поле `date` из формы. Несовпадение дат → проверка.
- Аванс: «новый сотрудник» = первая смена строго ПОСЛЕ 15-го числа → аванс 0.
- Везде строки `YYYY-MM-DD`; сравнение дат идёт лексикографически по строкам — формат
  обязан быть с ведущими нулями (`String(month).padStart(2,'0')`).

## Авторизация / логин
- Имя при логине нормализуется ё→е (`normalizeYo`), сравнение регистронезависимое по
  полному совпадению. «Семён» и «Семен» — одно и то же.
- Старший продавец определяется по `id`, начинающемуся с `SProd` (`isSeniorSeller`),
  ставка фикс 1300 и магазин может быть не привязан.
- В payroll-расчётах роли admin/accountant и пустые fullname ИСКЛЮЧАЮТСЯ из выборки.

## Роуты
- `POST /save-universal-corrections` объявлен ДВАЖДЫ. Express берёт первый совпавший —
  работает защищённый (стр.~2349); незащищённый дубль (стр.~2408) сейчас мёртв, но
  при перестановке/рефакторинге может «ожить» как дыра. Удалить дубль осознанно.
- lookup.js подключён на `app.use('/', ...)` — его пути живут в общем неймспейсе с
  server.cjs. Следить за коллизиями имён путей.

## Защита ручных правок
- При повторном `calculate-final-payroll` записи с `is_manual_adjustment`/`is_fixed`/
  `is_remainder_adjusted`/`is_termination` НЕ пересчитываются автоматикой. Если «расчёт не
  обновляет суммы» — скорее всего стоит один из этих флагов.

## Инфраструктура
- `withLock` — in-memory: на нескольких инстансах Fly.io НЕ защищает от гонок.
- Обработка `error.code === '42P01'` (relation does not exist) местами считается нормой —
  значит часть таблиц может отсутствовать; не принимать отсутствие за «ОК» вслепую.
- service_role_key → RLS не действует; авторизация только на уровне Express.
