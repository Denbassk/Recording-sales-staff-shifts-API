# mem:payroll_new_rules_2026 — Новые правила начисления ЗП продавцам (с 16.07.2026)

СТАТУС: рефакторинг в работе (июль 2026). Действуют **С 16 ИЮЛЯ 2026**.
До 15.07.2026 включительно — старые правила (`mem:business_logic §1`).
**Старших продавцов (`SProd*`) и магазин «Полевая» (store_id 38) НЕ трогаем** — у них всё по-старому.

Источник требований: постановка задачи владельца (dan) + офлайн-скрипты:
- `merge_reports.py` — вычитание никотина (по префиксам названий).
- `Просчет_бонусов_кулинария_пакеты_кофе.py` (`D:\CULINARY\...март_2026\`) — кулинария/пакеты/кофе.
Многие детали (штрих-коды, деление на смену) взяты из второго скрипта — он источник истины по Блоку 3.

## Переход старое → новое
- Смена с датой **≤ 2026-07-15** → старая формула (975/825 + бонус 5–12 грн/тыс сверх 13000).
- Смена с датой **≥ 2026-07-16** → новая формула (плавающая ставка + % + доп.бонусы).
- В коде уже есть `isNewRules = date >= 2026-07-01`, но он ТОЛЬКО для «Полевой». Для общих
  правил нужен ОТДЕЛЬНЫЙ порог `2026-07-16`; логику «Полевой» и старших не менять.
- Аванс за июль (период 1–15) целиком по старым правилам — новые правила туда не попадают.
  С августа аванс (1–15) уже по новым, но % и бонусы к 15-му ещё не финальны — см. «Архитектура».

## Блок 1. Ставка (плавающая, по магазину)
- По СРЕДНЕДНЕВНОМУ выторгу магазина за **предыдущий** месяц.
- Среднедневной = Σ `daily_revenue.revenue` за пред. месяц ÷ число **отработанных** дней
  (дни с кассой > 0). Магазин работает каждый день.
- Сигареты В выторг ВХОДЯТ (`daily_revenue.revenue` = полная касса).
- Пороги (грн/смена):
  - выторг **≤ 40 000 → 800** (ровно 40 000 → 800)
  - **40 000 < выторг ≤ 50 000 → 900** (ровно 50 000 → 900)
  - выторг **> 50 000 → 1000**
- Привязана к МАГАЗИНУ. Продавец в разных магазинах — берётся среднее того магазина,
  где стоит смена в этот день.
- Июль (дни 16–31) → по июню; август → по июлю; и т.д.
- Заменяет старое «975 (1 прод.) / 825 (2+)».

## Блок 2. Процент от продаж (дневной, делится на смену)
- Заменяет старую бонус-таблицу (5–12 грн/тыс).
- База = касса магазина за день − продажи товаров поставщика **ДЛ Солюшн** за этот день
  (табак + аксессуары + зажигалки).
- Пул на смену = **3% от базы**, делится поровну между продавцами на смене (каждому 3%/N):
  - 1 продавец → 3%; 2 → по 1,5%; 3 → по 1%.
- Считается по каждой смене/дню.

### ДЛ Солюшн — источник и метод (ПОДТВЕРЖДЕНО в BigQuery)
- Поставщик в приходах: `incoming_transactions.supplier = 'ДЛ Солюшн'` (255 штрих-кодов,
  приходы 2026-01…07). Продажи (`turnover_transactions`) поставщика НЕ содержат — только приходы.
- Метод: берём штрих-коды ДЛ Солюшн из приходов → находим их продажи (`check_amount`) в
  `turnover_transactions` по магазину/дню → вычитаем из кассы.
- ⚠️ РИСК подтверждён замером (июнь 2026): если брать штрих-коды ТОЛЬКО из приходов месяца
  (171 шт.) — теряем 2,56% продаж ДЛ (≈297 тыс. грн) и 60 проданных штрих-кодов, которые
  продавались, но в этом месяце не приходили.
- РЕКОМЕНДАЦИЯ: держать **накопительный справочник штрих-кодов ДЛ Солюшн** (union по ВСЕЙ
  истории приходов, а не за один месяц), обновлять помесячно. Полный набор (255) ловит все
  11,6 млн грн продаж ДЛ за июнь. Дополнительно: подстраховка по названиям сигарет
  (`Сигарети`, `Безтютюн.Нікотин.Под.`) на случай новых штрих-кодов, ещё не попавших в приходы.
  Полезно вести таблицу-справочник (Supabase или BQ) `dl_solution_barcodes`.
- Есть также `barcode_supplier_override` (12 строк) и `supplier_mapping` (156) — учесть
  ручные переопределения поставщика при сборке справочника.

## Хранение и влияние на выплаты
- Новые % и доп.бонусы — ЧАСТЬ зарплаты, облагаются налогом (ФОТ 22% как обычно).
- Должны попадать в gross (`payroll_calculations.total_pay` или доп.колонки), чтобы влиять
  на аванс/финал/ФОТ.

## Архитектура интеграции (предложение — на согласовании)
Гибрид: ставка — live, процент и бонусы — месячный батч (т.к. набор ДЛ Солюшн и списания
финальны только по закрытии месяца).
1. Дневной `/calculate-payroll`: считает СТАВКУ (по пред. месяцу, per store) сразу → в
   `payroll_calculations`.
2. Месячный батч (по закрытии месяца, когда BQ+списания готовы): считает per-shift процент
   (касса − ДЛ Солюшн) и 3 бонуса, делит на продавцов, пишет в Supabase (доп.колонки в
   `payroll_calculations` или отдельная таблица `shift_extra_bonuses` по employee_id+work_date).
   `total_pay = ставка + процент + бонусы`; далее аванс/финал/ФОТ работают как есть.
3. Реализацию батча начать с переиспользования готовых Python-скриптов (они уже кодируют
   штрих-коды, исключения «чай»/«Випічка», матч по алиасам), добавив запись результата в
   Supabase вместо только Excel. Позже — порт в Node (`@google-cloud/bigquery`).
