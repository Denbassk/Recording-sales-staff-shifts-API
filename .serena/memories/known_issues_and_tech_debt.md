# mem:known_issues_and_tech_debt — техдолг и риски (честно)

## Безопасность (высокий приоритет)
1. **Эндпоинты без полной авторизации** (нарушение инварианта №1 из `mem:core`):
   - `GET /employees` — вообще без checkAuth, отдаёт ФИО всех активных сотрудников.
   - `GET /api/employees-with-limits` — checkAuth без роли (seller увидит лимиты всех).
   - `GET /api/get-employee-card-limit/:employee_id` — checkAuth без роли.
   - `POST /save-universal-corrections` зарегистрирован ДВАЖДЫ; второй (стр.~2408) БЕЗ
     auth. Сейчас затенён первым, но при перестановке роутов станет открытой дырой.
2. **Пароли в открытом виде**: `employees.password` сравнивается равенством, без хеша.
   Логин по точному паролю + имя. Нет соли/bcrypt, нет rate-limit на /login.
3. **CORS `origin: true` + credentials** — отражает любой Origin с куками; в связке с
   httpOnly-cookie это ослабляет защиту от CSRF (особенно sameSite='lax' вне prod).
4. **service_role_key на сервере** — любая дыра в роутере = доступ ко всем данным (RLS off).
5. Рассинхрон жизни сессии: JWT 8ч, cookie maxAge 7 дней.

## Архитектура / поддерживаемость
6. **`server.cjs` ~4150 строк — монолит**: ~50 эндпоинтов + бизнес-логика + хелперы в
   одном файле. Тяжело тестировать и ревьюить. Кандидат на разбиение по модулям
   (payroll, advances, card-limits, fot, backups, auth).
7. **Дублирование auth-слоя**: server.cjs (`checkAuth`/`checkRole`) vs lookup.js
   (`checkAuthCookie`/`requireAdmin`) — две независимые реализации, легко разойтись.
8. **In-memory `withLock`** (Map в процессе) — НЕ масштабируется на несколько инстансов
   Fly.io; при автоскейле возможны гонки в `calculate-final-payroll`. Нужен распределённый
   лок (advisory lock Postgres / отдельная таблица-лок).
9. **Дублирование supabase-клиента**: в lookup.js `createClient` вызывается ВНУТРИ каждого
   хэндлера заново (а не один раз на модуль).
10. **Куча backup-файлов в репо** (`server_backup_*`, `server_original.cjs`,
    `public/payroll_backup*`, `script_backup.js`, `.rar`/`.zip`) — мусор, путает навигацию.

## Финансовая корректность
11. **`validatePayrollCalculation`: проверка лимита карты ЗАКОММЕНТИРОВАНА** — функция не
    ловит превышение лимита; полагаемся только на логику распределения. Раскомментировать
    нельзя как было (жёсткие 8700), нужна проверка против индивидуального `cardLimit`.
12. **ФОТ хардкодит `cardLimit = 16000`** в `buildFotReport`, игнорируя `card_limit_types`.
    Для сотрудников со STANDARD (8700) налоговая база на карту считается завышенно.
13. `logFinancialOperation` вызывается не во всех денежных операциях (аванс-фиксации,
    изменения лимитов) — пробелы в аудите.
14. Местами ловится Postgres `42P01` (таблица не существует) как норма — намёк, что часть
    таблиц могла не примениться миграцией; состояние схемы на проде стоит сверить.

## Тестирование / процессы
