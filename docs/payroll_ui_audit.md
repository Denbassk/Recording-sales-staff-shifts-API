# Аудит интерфейса расчёта зарплаты + план правок

Файлы: `public/payroll.html` (1225 стр.), `public/payroll.js` (5372 стр.),
эндпоинты `server.cjs`: `/calculate-final-payroll` (1913), `/fix-advance-payment` (1195),
`/cancel-advance-payment` (1430), `/fix-manual-advances` (1566), `/adjust-final-payment` (2187).

## 1. КОРЕНЬ «сбоя после фиксации» (главное)
Сейчас расчёт живёт в ДВУХ независимых местах, и они расходятся:

- **Браузер** (`recalculateRow`, payroll.js): при правке премии/штрафа/недостачи пересчитывает
  распределение «карта/наличные» ПРЯМО В ТАБЛИЦЕ, но:
  - использует захардкоженный лимит карты `16000` как fallback (`dataset.cardLimit || 16000`),
    а у сотрудника может быть 8700;
  - это распределение **никуда не сохраняется** — `saveAdjustments` шлёт на сервер только
    `manual_bonus/penalty/shortage`, без карты/налички.
- **Бэкенд** (`/calculate-final-payroll`): при «Итоговом расчёте» пересчитывает карту/наличные
  С НУЛЯ по своим правилам (индивидуальный лимит, диф-логика `is_remainder_adjusted`, округления,
  «тихая» корректировка налички при расхождении > 1 грн, строки 2099–2106).

Итог: бухгалтер видит в таблице одно число, в БД сохраняется другое. После фиксации и правок
расхождение накапливается → приходится править вручную.

### Почему именно ПОСЛЕ фиксации
`is_fixed` защищает от пересчёта только **аванс** (server.cjs 2010–2022). А **остаток**
(карта/наличные) защищён, только если отдельно стоит `is_remainder_adjusted` (2034–2076).
Т.е. сценарий: зафиксировали выплату → поменяли недостачу → «Итоговый расчёт» → аванс сохранён,
а остаток пересчитан заново и перезатёрт. Половина строки «заморожена», половина — нет. Это и есть баг.

## 2. Проблемы логики
1. **Два движка расчёта** (браузер + сервер) — рассинхрон (см. §1).
2. **«Полу-защита» после фиксации**: `is_fixed` не защищает распределение остатка.
3. **Клубок флагов** `is_fixed / is_manual_adjustment / is_remainder_adjusted / is_termination` —
   каждый защищает своё поле, взаимодействия сложны, легко получить противоречивое состояние.
4. **Статус «зафиксировано» определяется по эмодзи** `🔒` в `innerHTML` ячейки
   (`fixAdvancePayment`, 2715) — не по данным. Хрупко.
5. **«Тихая» корректировка** налички при несходимости математики (2102–2106) — маскирует ошибку,
   вместо того чтобы показать её бухгалтеру.
6. **Дубликат функции** `recalculateRow` (payroll.js 1160 и 1784 — почти идентичны, вторая
   переопределяет первую). Правки легко разъезжаются.
7. **Проверка лимита карты закомментирована** (`validatePayrollCalculation`, см. `mem:known_issues`) —
   при распределении можно превысить лимит и не заметить.
8. **Хардкод лимита 16000** в браузере и в ФОТ (`buildFotReport`) — расходится с индивидуальными.

## 3. Проблемы интерфейса/оформления
1. **Перегруженная вкладка «Отчёт за месяц»**: в один экран свалено 12+ кнопок
   (Сформировать, Аванс, Итоговый расчёт, Зафиксировать выплату, Зафиксировать ручные
   корректировки, Отменить, История, Экспорт, Печать, Ведомость наличных, Очистить БД, Выход).
2. **Две почти одинаковые кнопки** «🔒 Зафиксировать выплату» и «🔧 Зафиксировать ручные
   корректировки» — бухгалтеру неясно, что нажимать и чем отличаются.
3. **Нет наглядного порядка действий**. Правильная последовательность (выручка → зарплата →
   отчёт → аванс → фиксация → правки → итог) нигде не показана; легко сделать не по порядку.
4. **Статусы строки не видны**: нельзя с ходу понять, где черновик, где зафиксировано,
   где правили вручную.
5. **Опасное рядом с рутинным**: «🗑️ Очистить БД» стоит рядом с обычными кнопками.
6. **Визуальный шум**: градиентные кнопки с тенями текста, много эмодзи, инлайн-стили по всему
   HTML (1225 строк), разнобой в оформлении.
7. **Нет индикатора «несохранённые изменения»** — правки сохраняются по таймеру (debounce 800 мс),
   момент сохранения не виден.

## 4. Предлагаемые правки — ЛОГИКА (приоритет: убрать сбой)
1. **Единый источник правды = бэкенд.** Браузер перестаёт сам считать карту/наличные. После правки
   строки вызываем серверный «предпросчёт» (новый эндпоинт `/preview-final-row` или переиспользуем
   расчёт) и показываем ЕГО числа. Ни одного числа, посчитанного только в браузере.
2. **Чёткая модель блокировки строки** вместо клубка флагов:
   - `draft` (черновик) → правки свободны, всё пересчитывается;
   - `locked` (зафиксировано) → строка неизменна; «Итоговый расчёт» её НЕ трогает вообще
     (не половину, а целиком);
   - чтобы изменить зафиксированную строку — явная кнопка «Разблокировать для правки»
     (делает бэкап → снимает lock → пересчитывает строку ЦЕЛИКОМ заново → бухгалтер правит → снова фиксирует).
   Это убирает сценарий «половина заморожена, половина пересчитана».
3. **`is_fixed` защищает строку целиком** (аванс + распределение остатка), а не только аванс.
4. **Убрать «тихую» корректировку**: при несходимости — не молча менять наличные, а подсветить строку
   и показать предупреждение бухгалтеру.
5. **Статус фиксации — из данных** (`is_fixed`/статус строки), а не из эмодзи в HTML.
6. **Удалить дубликат** `recalculateRow`; использовать индивидуальный лимит карты, убрать хардкод 16000.
7. **Вернуть проверку лимита карты** (раскомментировать/переписать `validatePayrollCalculation`
   против индивидуального `cardLimit`).

## 5. Предлагаемые правки — ИНТЕРФЕЙС (проще для бухгалтера)
1. **Мастер по шагам** на вкладке расчёта: 1) Выручка → 2) Зарплата по дням → 3) Отчёт за месяц →
   4) Аванс → 5) Бонусы (пересчёт extras) → 6) Итоговый расчёт → 7) Фиксация/выплата. Текущий шаг
   подсвечен, следующие недоступны, пока не выполнен предыдущий.
2. **Одна понятная кнопка фиксации** + явные состояния строки: `Черновик` / `🔒 Зафиксировано` /
   `✏️ Изменено вручную`. Вторую кнопку («ручные корректировки») убрать или объединить.
3. **Бейдж-статус в каждой строке** (цветной) — сразу видно состояние.
4. **Индикатор несохранённого** + явная кнопка «Сохранить», плюс подтверждение «сохранено».
5. **Опасные действия отдельно** (Очистить БД — в «Настройки», с двойным подтверждением и вводом слова).
6. **Единый стиль**: вынести CSS из инлайна, спокойные кнопки без градиент-теней, меньше эмодзи,
   понятные подписи. HTML/JS разбить на модули (по вкладкам).
7. **Кнопка «Пересчитать бонусы за период»** (вызов `/calculate-payroll-extras`) — сюда же, шаг 5.
8. **Переименование столбцов** под новые параметры (Блоки 1–3): показать ставку, процент,
   пакеты/кофе/кулинарию, аванс, остаток карта/наличные, к выплате. Точные названия согласуем.
