# 03. Система возвратов товара

## Назначение

Продавец в магазине сканирует штрих-коды возвращаемых товаров прямо с кассы. Система:
- ищет товар в BigQuery (склад конкретного магазина → сеть → каталог → неизвестный)
- формирует «корзину возврата»
- сохраняет возврат в Supabase
- администраторы видят все возвраты, фильтруют и экспортируют в Excel

## Интерфейс продавца (`/returns.html`)

### Доступ
- Только сотрудник с **активной сменой** в магазине
- Если нет смены → страница не открывается (показывает ошибку)
- Администраторы **не могут** оформлять возвраты

### Запуск на кассе
Файл `Returns.bat`:
- Читает `device.json` → берёт `device_key`
- Открывает `https://shifts-api.fly.dev/returns.html?device={DEVICE_KEY}` в Chrome или Edge

### Поиск товара — 2 способа

**1. По штрих-коду (основной)**
- Поле `#barcodeInput` всегда в фокусе — сканер сразу читает
- Ввод через Enter → `handleScan(barcode)`
- Валидация: 6–14 цифр

**2. По названию (резервный)**
- Поле `#searchInput` — debounce 300мс, минимум 3 символа
- Запрос к `/search-product?q=...&store_address=...`

### Цветовые статусы (плашки)

| Цвет | Значение | Действие |
|------|----------|----------|
| 🟢 Зелёная | Товар **на складе магазина** | Окно количества → добавить |
| 🟠 Оранжевая | Товар есть **в сети**, но не у нас | Подтверждение → добавить |
| 🔵 Синяя | Товар в **каталоге**, нигде нет на складе («хвост») | Подтверждение → добавить |
| 🟡 Жёлтая | **Неизвестный штрих-код** | Ручной ввод: название + цена + кол-во |

### Модальные окна
- `#qtyModal` — ввод количества для зелёного товара (кнопки +/−, поддержка весовых штрих-кодов, начинающихся на «2»)
- `#confirmModal` — подтверждение для оранжевого и синего
- `#manualModal` — ручной ввод для жёлтого (без штрих-кода в базе)
- `#helpModal` — справка (кнопка «❓ Допомога» в шапке)

### Корзина
- Хранится в `localStorage` под ключом `returnsCart` (восстанавливается при перезагрузке)
- Каждая позиция: `{ barcode, product_name, cost_price, quantity, lookup_status, stock_at_scan, _id }`
- Редактирование количества прямо в строке корзины
- Кнопка ✕ — удалить позицию
- Кнопка «Очистити» — очистить всю корзину
- Показывает счётчик «Не з бази» (жёлтых позиций)

### Отправка возврата
- POST `/returns` → тело: `{ items: [...] }`
- После успеха: `toast("✅ Повернення RET-xxxxxx-xxx відправлено")`, очистка корзины
- Номер формата `RET-YYMMDD-NNN` генерируется через Supabase RPC `generate_return_number`

### Звуки (Web Audio API)
| Статус | Частота | Форма |
|--------|---------|-------|
| green | 880 Hz | sine |
| orange/blue | 660 Hz | sine |
| yellow | 330 Hz | square |
| error | 200 Hz | sawtooth |
| submit | 880 + 1175 Hz | sine×2 |

## Интерфейс администратора (`/admin-returns.html`)

### Доступ
- Только роли: `admin`, `accountant`, `curator`
- Проверка через `/me-with-store`

### Возможности
- **Фильтры:** период (от/до), магазин, статус (активные/архив/все), цвет статуса ШК
- **Авто-обновление** каждые 30 секунд (индикатор ● анимирован)

### Вкладки
| Вкладка | Содержимое |
|---------|------------|
| Список повернень | Карточки возвратов с разворачиваемыми позициями |
| Зведення: постачальник | Группировка по первому слову названия |
| Зведення: SKU | Группировка по штрих-коду + название |
| Зведення: магазин | Сумма/кол-во по каждому магазину |
| 🟡 Не з бази | Таблица всех жёлтых позиций (неизвестные ШК) |

### Статистика
В шапке вкладки «Список»: кол-во возвратов, кол-во позиций, общая сумма.

### Архивирование
- Кнопка «В архів» на карточке → POST `/returns/:id/archive`
- Устанавливает `archived_at`, `processed_by`, `processed_at`, `status = 'archived'`

### Экспорт в Excel (XLSX)
Кнопка «📥 Експорт» — генерирует файл прямо в браузере через **ExcelJS** + **FileSaver**.

7 листов:
1. **Зведення** — сводная: период, кол-во, топ-10 поставщиков, топ-10 магазинов
2. **Возвраты** — шапки всех возвратов
3. **Позиції** — все позиции развёрнуто (с цветовой заливкой статуса)
4. **По постачальнику** — агрегация по поставщику
5. **По магазину** — агрегация по магазину
6. **По SKU** — агрегация по штрих-коду
7. **Не з бази** — только жёлтые позиции

## API-эндпоинты возвратов (в `routes/lookup.js`)

### GET `/lookup?barcode=&store_address=`
- Требует JWT cookie
- Ищет в BigQuery: `stock_current` (текущий склад) + `barcode_catalog`
- Учитывает алиасы магазина (таблица `store_address_aliases`)
- Возвращает: `{ success, color, barcode, product_name, cost, retail_price, stock_in_store, ... }`

### GET `/search-product?q=&store_address=`
- Полнотекстовый поиск по названию через BigQuery LIKE
- Возвращает до 25 результатов: 15 green + 15 orange + 10 blue

### GET `/me-with-store`
- Определяет магазин текущего пользователя: замена → смена → постоянная привязка
- Администраторы получают `store: null`

### POST `/returns`
- Тело: `{ items: [{ barcode, product_name, quantity, cost_price, lookup_status }] }`
- Создаёт запись в `returns` + позиции в `return_items`
- Генерирует `return_number` через RPC `generate_return_number`

### GET `/returns?from=&to=&store_id=&status=&lookup_status=&archived=`
- Только admin/accountant/curator
- Возвращает возвраты с вложенными позициями (`select('*, return_items(*)')`)
- Лимит 500 записей

### GET `/returns/summary?group_by=supplier|sku|store&from=&to=`
- Агрегирует позиции по группе
- Возвращает: `{ key, qty, total, returns_count }`

### POST `/returns/:id/archive`
- Архивирует возврат

### GET `/stores`
- Список активных магазинов для фильтров

## BigQuery (источник данных)

Проект: `family-market-analytics`  
Dataset: `returns_system`

| Таблица | Описание |
|---------|----------|
| `stock_current` | Текущие остатки: barcode, store_address, product_name, qty_in_stock, cost_price, retail_price |
| `barcode_catalog` | Каталог всех известных штрих-кодов: barcode, product_name, last_cost_price, last_retail_price |

## Supabase (транзакционная база)

| Таблица | Описание |
|---------|----------|
| `returns` | Шапки возвратов: id, return_number, employee_id, store_id, store_address, status, items_count, total_cost, notes, archived_at, processed_by, processed_at |
| `return_items` | Позиции: return_id, barcode, product_name, quantity, cost_price, lookup_status, reason |
| `stores` | Магазины |
| `store_address_aliases` | Исторические названия адресов магазинов |
| `employees` | Сотрудники |
| `shifts` | Смены |
| `substitutions` | Подмены (работа в чужом магазине) |
| `employee_store` | Постоянная привязка продавца к магазину |

RPC функция: `generate_return_number` — генерирует уникальный номер вида `RET-YYMMDD-NNN`

## Еженедельный бэкап

**Файл:** `scripts/backup-returns.js`  
**Триггер:** каждое воскресенье в 20:30 UTC (`cron: '30 20 * * 0'`)  
**Что делает:**
1. Читает последние 5000 возвратов из Supabase
2. Строит XLSX (7 листов, аналогично экспорту из браузера)
3. Загружает в Google Drive (папка `DRIVE_BACKUP_FOLDER_ID`)
4. Удаляет старые копии, оставляет последние 10

**Переменные окружения (GitHub Secrets):**
- `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`
- `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REFRESH_TOKEN`
- `DRIVE_BACKUP_FOLDER_ID`

## Известные особенности и нюансы

- **Весовые штрих-коды:** EAN-13, начинаются на «2» → шаг количества 0.001 (кг)
- **Алиасы магазинов:** если магазин переименовался, его старые названия хранятся в `store_address_aliases` — BigQuery ищет по всем активным алиасам сразу
- **Возврат фокуса:** после закрытия любого модального окна фокус автоматически возвращается в поле штрих-кода
- **Резервирование корзины:** `localStorage.returnsCart` — при перезагрузке страницы корзина восстанавливается
- **Лимит запросов:** GET `/returns` — максимум 500 за один запрос
- **Администраторы не оформляют возвраты** — интерфейс `/returns.html` блокируется для ролей без магазина
